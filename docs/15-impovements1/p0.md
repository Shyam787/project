# P0 - Core SaaS Transformation, Authentication, Organization Management, Document Governance, and UX Foundation

## Objective

This phase transforms the current Enterprise RAG application from a demo-oriented single-tenant implementation into a dynamic multi-tenant SaaS platform.

P0 is the highest priority phase.

No P1 or P2 work may begin until all P0 requirements are implemented, validated, and functioning correctly.

---

# Critical Non-Negotiable Rules

## Rule 1: Remove Hardcoded Demo Behavior

The current implementation contains hardcoded demo concepts.

Examples include:

* ACME
* tenant-acme
* demo-admin
* demo-viewer
* static tenant assumptions
* static role assumptions

These must be removed from production workflows.

No production page may display:

```text
demo-admin
demo-viewer
tenant-acme
ACME Corporation
```

unless explicitly operating inside the isolated Demo Workspace defined in later phases.

All production behavior must become dynamic.

---

## Rule 2: Preserve Existing Working RAG Functionality

The following must continue working throughout implementation:

* document upload
* chunking
* indexing
* retrieval
* RBAC
* citations
* hallucination scoring
* Qdrant integration
* PostgreSQL integration
* Keycloak integration

Do not rewrite functioning subsystems unnecessarily.

Improve them incrementally.

Required fields:

```text

Organization ID

Confirm Password
```

---

## Validation Rules

### Organization Name

Required.

Minimum:

```text
3 characters
```

Maximum:

```text
100 characters
```

---

### Organization ID

Required.

Must be globally unique.

Allowed:

```text
a-z
0-9
hyphen
```

Examples:

```text
acme
globex
wayne-industries
```

Rejected:

```text
ACME
Acme Corp
acme!
```

---

### Password Requirements

Password requirements must be visible before submission.

Display live validation feedback.

Requirements:

```text
Minimum 12 characters
At least one uppercase letter
At least one lowercase letter
At least one number
At least one special character
```

Example UI:

✓ Minimum 12 characters

✓ Contains uppercase letter

✓ Contains lowercase letter

✓ Contains number

✓ Contains special character

---

# Authentication Architecture

## Remove Organization Login

Do NOT implement a separate organization login page.

The final architecture is:

```text
Create Organization
↓
Create Initial Administrator
↓
Administrator Login
↓
Administrator Creates Users
↓
Users Login
↓
Tenant Context Loaded Automatically
```

Organization context must be derived from authenticated user identity.

---

## User Login

Login fields:

```text
Email
Password
```

only.

No:

```text
Organization ID
Tenant ID
Workspace ID
```

must be entered by users.

---

## Keycloak Tenant Resolution

Authenticated user tokens must contain:

```text
tenant_id
roles
email
```

The backend must derive organization context from these claims.

The frontend must never determine tenant identity itself.

# Document Lifecycle

Implement:

```text
Preview
Download
Archive
Restore
Soft Delete
Hard Delete
```

---

## Archive

Document remains searchable history-wise but is excluded from active usage.

---

## Restore

Restores archived document.

---

## Soft Delete

Document hidden from normal usage.

Recoverable.

---

## Hard Delete

Permanent removal.

Must remove:

```text
Database metadata
Chunks
Vectors
Stored file
```

No orphaned data allowed.

---

# Local File Storage

Do not use Cloudinary.

Store files locally.

Create:

```text
storage/
└── tenants/
```

Example:

```text
storage/
└── tenants/
    ├── acme/
    ├── globex/
    └── company-x/
```

Uploaded files must persist through:

```text
Logout
Login
Browser refresh
Application restart
Docker restart
```

until explicitly removed.

---
